Complexity Reference

Big-O describes how an algorithm's cost grows as input size n grows toward infinity — it deliberately ignores constants and lower-order terms, because those stop mattering at scale.

How to compute it

1Count the basic operations as a function of n. A single loop over n items is n operations.
2Drop constant factors: 3n and n are both O(n) — the constant doesn't change the growth shape.
3Keep only the dominant term: n² + n simplifies to O(n²), since the quadratic term swamps the linear one as n grows.
4Nested loops multiply: a loop inside a loop, each running n times, gives O(n²).

Best, average, worst case

These describe different input scenarios, not different algorithms. Quicksort averages O(n log n) across random inputs, but degrades to O(n²) on an already-sorted array if the pivot is chosen poorly (see [09]). Cheat sheets usually quote average case unless noted.

Space complexity

The same rules apply to memory: count the extra memory an algorithm uses beyond the input itself. A recursive function's call stack counts — recursing n levels deep uses O(n) space even if no array is ever allocated.

Arrays & Strings

The default container, and the one whose costs are easiest to derive directly from how memory actually works.

Why access is O(1)

An array is one contiguous block of memory. To read index i, the runtime computes address = base + i × elementSize — a single arithmetic operation, regardless of how large the array is. That's the entire reason arrays are fast: there's no searching, just math.

Why insert/delete is O(n)

Contiguity is also the catch. Inserting at index 0 means every element after it has to physically shift one slot to the right to preserve the "no gaps" property — that's up to n moves.

HOW ARRAYLIST GETS AMORTIZED O(1) APPEND

An ArrayList isn't magic — under the hood it's a plain array with spare capacity. Most add() calls just drop the element into unused space: O(1). When the backing array fills up, it allocates a new array at double the size and copies everything over: one expensive O(n) operation. Because doubling happens exponentially less often as the list grows, that occasional cost spreads out to O(1) on average per insert — this is what "amortized" means.

Basics.java
int[] arr = {1, 2, 3};
List<Integer> list = new ArrayList<>();
list.add(5); list.get(0); list.remove(0);

int[][] grid = new int[rows][cols];      // 2D array

StringBuilder sb = new StringBuilder();
sb.append("a").reverse().toString();

Arrays.sort(arr);                          // primitives, ascending only
Integer[] boxed = {3, 1, 2};
Arrays.sort(boxed, Collections.reverseOrder());
WHY STRINGBUILDER EXISTS

Java strings are immutable — s += "x" doesn't mutate s, it allocates an entirely new string and copies the old contents into it. Do that inside a loop of n concatenations and you've paid for 1 + 2 + 3 + … + n characters copied: O(n²) total. StringBuilder keeps one mutable internal buffer and only resizes it the same way ArrayList does, bringing the whole loop down to O(n).

Worked example — two pointers, palindrome check

Checking if "racecar" is a palindrome without extra memory:

1Place left at index 0 and right at index 6 (the last index).
2Compare s[left] and s[right]: 'r' == 'r' → match, move both pointers inward.
3Repeat: 'a'=='a', 'c'=='c', pointers keep closing in.
4Pointers meet or cross at the middle ('e') without ever mismatching → palindrome confirmed in a single O(n) pass, O(1) extra space.

Related patterns: sliding window and prefix sums use the same "walk the array once, track state" idea — see [14].

Linked Lists

Every node only knows about the node after it (or before it, for doubly-linked). That single design choice flips every cost in [01] upside down.

Why the trade-off flips

There's no formula for "the address of node 5" the way there is for arrays — you have to walk from the head, following pointers one at a time, which makes access O(n). But inserting a node is just rewiring two pointers wherever you're already standing: O(1), no shifting required.

VariantStructureNotable trait
Singly linkednode → nextCan't walk backward; lowest memory overhead
Doubly linkedprev ← node → nextO(1) removal given just a node reference
Circulartail → headNo null terminator; used for round-robin scheduling
ListNode.java
class ListNode {
    int val;
    ListNode next;
    ListNode(int val) { this.val = val; }
}

ListNode reverse(ListNode head) {
    ListNode prev = null;
    while (head != null) {
        ListNode next = head.next;  // save before we overwrite it
        head.next = prev;           // flip the pointer
        prev = head;
        head = next;
    }
    return prev;
}

Worked example — Floyd's cycle detection

Two pointers, one moving twice as fast as the other:

1slow and fast both start at head.
2Each step, slow moves 1 node, fast moves 2 nodes.
3If there's no cycle, fast hits null and exits — no cycle exists.
4If there is a cycle, fast enters it first and laps slow from behind. Since the gap between them shrinks by 1 every step, they're guaranteed to land on the same node eventually — O(n) time, O(1) space, no extra memory needed to track "visited."
WHY A DUMMY HEAD NODE HELPS

A common pain point: deleting the actual first node needs special-case code, because there's no "previous" node to repoint. Allocating a fake dummy node before the real head means every real node — including the first — has a predecessor, so insert/delete logic stays uniform with no edge cases.

Stacks & Queues

Same underlying idea — a list with restricted access — but opposite rules about which end you're allowed to touch.

How they relate to recursion

Every function call you make is secretly pushed onto a real call stack: local variables, the return address, all of it. That's why recursion that goes too deep throws a StackOverflowError — it's the same stack you're modeling explicitly with Deque.

Stack — LIFO
Deque<Integer> stack = new ArrayDeque<>();
stack.push(1);
stack.pop();
stack.peek();
Queue — FIFO
Queue<Integer> q = new LinkedList<>();
q.offer(1);
q.poll();
q.peek();

Worked example — monotonic stack (next greater element)

For [2, 1, 2, 4, 3], find the next number to the right that's bigger than each element:

1Walk left to right, keeping a stack of indices whose "next greater" isn't found yet.
2At each new number, pop every stacked index whose value is smaller than the current one — the current number is their answer.
3Push the current index, then continue. Because each index is pushed once and popped at most once, the whole pass is O(n) — not O(n²), despite the nested-looking pops.
ARRAY-BACKED QUEUES NEED WRAP-AROUND

If you implement a queue on a plain array, removing from the front by shifting everything left is O(n) — defeating the point. A circular buffer instead tracks separate head and tail indices that wrap around the array's end, keeping both enqueue and dequeue O(1).

Use cases: stack → bracket matching, undo, DFS. Queue → BFS, task scheduling. Deque → sliding window maximum.

Hash Tables

The fastest "have I seen this before" available — provided the hash function does its job.

How it actually works

1A hash function converts a key into an integer (its hashCode()), then maps that integer to one of a fixed number of buckets, usually via hash % numBuckets.
2Lookup hashes the key, jumps straight to that bucket, and checks for a match — no scanning required, hence O(1).
3Two different keys can land in the same bucket — a collision. Java's HashMap handles this by chaining: each bucket holds a small list (or balanced tree, once a bucket gets large) of entries.
4As the map fills up, more collisions pile into each bucket. Once the ratio of entries to buckets (the load factor, default 0.75 in Java) is exceeded, the map resizes: allocates a bigger bucket array and rehashes everything — the same amortized-cost idea as ArrayList in [01].
WHY WORST CASE IS O(n)

If every key collided into a single bucket — a broken or adversarial hash function — lookup degrades into scanning a linked list: O(n). This is also why hashCode() and equals() must agree: if two objects are equals but have different hash codes, HashMap will look in the wrong bucket and never find the entry at all.

Lookup.java
Map<String, Integer> map = new HashMap<>();
map.put("a", 1);
map.getOrDefault("b", 0);
map.containsKey("a");

Set<Integer> set = new HashSet<>();
set.add(1); set.contains(1);

LinkedHashMap<String, Integer> ordered = new LinkedHashMap<>(); // insertion order
TreeMap<String, Integer> sorted = new TreeMap<>();               // sorted by key, O(log n) ops

Trees

A recursive idea: a tree is a node plus two smaller trees. Almost every tree algorithm is a direct translation of that definition into code.

Why inorder traversal gives sorted output for a BST

A binary search tree maintains one invariant at every node: everything in the left subtree is smaller, everything in the right subtree is larger. Inorder traversal visits left subtree → node → right subtree — which, by that invariant, visits smaller values, then the current value, then larger values. Apply that recursively at every node and the output comes out fully sorted, for free.

BinaryTree.java
class TreeNode {
    int val;
    TreeNode left, right;
    TreeNode(int val) { this.val = val; }
}

void inorder(TreeNode n, List<Integer> out) {
    if (n == null) return;
    inorder(n.left, out);
    out.add(n.val);
    inorder(n.right, out);
}

void levelOrder(TreeNode root) {           // BFS, level by level
    Queue<TreeNode> q = new LinkedList<>();
    if (root != null) q.offer(root);
    while (!q.isEmpty()) {
        TreeNode node = q.poll();
        if (node.left != null) q.offer(node.left);
        if (node.right != null) q.offer(node.right);
    }
}
WHY "BALANCED" MATTERS

O(log n) search assumes the tree's height is roughly log n. But insert values in already-sorted order — 1, 2, 3, 4… — into a plain BST, and every node only ever gets a right child. The tree degenerates into what is structurally a linked list, and search becomes O(n). Self-balancing trees (AVL, Red-Black) fix this by performing rotations after insert/delete that re-shuffle nodes to keep height logarithmic, guaranteeing the O(log n) bound regardless of insertion order.

Finding the lowest common ancestor

To find the LCA of two nodes: recurse into both children. If a node finds the target in its left subtree and its right subtree, that node itself is the LCA — it's the exact point where the paths to each target diverge.

Heaps / Priority Queues

A tree that only promises one thing — the smallest (or largest) item is always at the root — and is clever enough to store that tree in a plain array.

The array trick

A heap is always a complete binary tree (every level full except possibly the last, filled left to right), which means its shape is entirely predictable. That lets it live in a flat array with no pointers at all: for index i, the parent is at (i-1)/2, the left child at 2i+1, the right child at 2i+2.

Why insert/extract are O(log n)

1Insert: append the new value at the end of the array, then "sift up" — repeatedly swap it with its parent while it's smaller (min-heap), until the heap property holds.
2Extract min: swap the root with the last element, shrink the array by one, then "sift down" the new root — repeatedly swap with its smaller child until it settles.
3Both operations only ever travel along one path from root to leaf — the tree's height. Since the tree is complete, height is always log n, which is exactly the cost of each operation.
Heaps.java
PriorityQueue<Integer> minHeap = new PriorityQueue<>();
PriorityQueue<Integer> maxHeap = new PriorityQueue<>(Collections.reverseOrder());
PriorityQueue<int[]> pq = new PriorityQueue<>((a, b) -> a[0] - b[0]); // custom comparator

Use cases: k-th largest/smallest, top-k frequent, merge k sorted lists, Dijkstra's algorithm (next up in [08]).

Tries

A tree shaped exactly like the words stored in it — every shared prefix becomes a shared path, so the tree literally encodes the structure of the dictionary.

Worked example — inserting "cat"

1Start at the root. Look at child slot 'c' — empty, so create a new node there.
2Move into the 'c' node, look at slot 'a' — create it, move in.
3Repeat for 't', then mark that final node's isWord = true.
4Searching "ca" afterward will walk to a valid node — but isWord is false there, so it's a known prefix, not a complete word. That distinction is exactly what powers autocomplete.
Trie.java
class TrieNode {
    TrieNode[] children = new TrieNode[26];
    boolean isWord;
}

void insert(TrieNode root, String word) {
    TrieNode node = root;
    for (char c : word.toCharArray()) {
        int i = c - 'a';
        if (node.children[i] == null) node.children[i] = new TrieNode();
        node = node.children[i];
    }
    node.isWord = true;
}
THE TRADE-OFF

Lookup cost is O(k) where k is the word's length — independent of how many other words are stored. That's strictly better than a hash set's average O(k) hashing cost for prefix-style queries, but it costs memory: each node reserves space for up to 26 children whether they're used or not.

Graphs

Trees with the hierarchy constraint removed — any node can connect to any other, which is precisely why cycles and "is everything still connected" become real questions.

Representation trade-off

RepresentationSpaceEdge lookupBest for
Adjacency listO(V+E)O(degree)Sparse graphs (most real ones)
Adjacency matrixO(V²)O(1)Dense graphs, frequent edge checks

BFS vs. DFS — not interchangeable

BFS uses a queue and explores level by level, which is exactly why it's the one that finds the shortest path in an unweighted graph — it visits everything 1 edge away before anything 2 edges away. DFS uses a stack (or recursion) and commits to going as deep as possible before backtracking, which is what makes it the natural fit for cycle detection, connectivity checks, and topological sort.

Graph.java
Map<Integer, List<Integer>> graph = new HashMap<>();
graph.computeIfAbsent(u, k -> new ArrayList<>()).add(v);

void bfs(int start, Map<Integer, List<Integer>> graph) {
    Set<Integer> visited = new HashSet<>();
    Queue<Integer> q = new LinkedList<>();
    q.offer(start); visited.add(start);
    while (!q.isEmpty()) {
        int node = q.poll();
        for (int nb : graph.getOrDefault(node, List.of())) {
            if (visited.add(nb)) q.offer(nb);
        }
    }
}

Worked example — Dijkstra's shortest path

1Set every node's distance to infinity except the start node, which is 0. Put (0, start) in a min-heap keyed by distance.
2Pop the closest unvisited node from the heap. Because all edge weights are non-negative, the moment a node is popped, its distance is final — nothing visited later could offer a shorter path back to it.
3"Relax" each neighbor: if dist[current] + edgeWeight < dist[neighbor], update it and push the better distance onto the heap.
4Repeat until the heap is empty. Total cost: O((V+E) log V), dominated by heap operations.

Worked example — topological sort (Kahn's algorithm)

1Compute each node's in-degree (number of incoming edges).
2Queue up every node with in-degree 0 — they have no unmet dependencies.
3Pop a node, add it to the order, then decrement the in-degree of each of its neighbors. Any neighbor that drops to 0 joins the queue.
4If every node makes it into the final order, it's valid. If some are left stranded with in-degree > 0, the graph has a cycle and no valid order exists.
WHY UNION-FIND IS NEARLY O(1)

Naively, "are these two nodes connected" needs a full traversal. Union-Find instead has each node point toward a representative "root" of its component. Path compression flattens the pointer chain every time you look something up, and union by rank always attaches the smaller tree under the larger one — together they keep the trees so shallow that each operation runs in amortized near-constant time, not the naive O(n).

Every sorting question in an interview is really a question about which trade-off you're willing to accept — there's no free option.

WHAT "STABLE" ACTUALLY MEANS

A stable sort keeps equal elements in their original relative order. It matters when you sort by a second key after already sorting by a first — say, sorting people by last name after they're already sorted by first name. An unstable sort can silently scramble that earlier ordering.

QuickSort.java
void quickSort(int[] a, int lo, int hi) {
    if (lo >= hi) return;
    int pivot = a[hi], i = lo;
    for (int j = lo; j < hi; j++) {
        if (a[j] < pivot) { swap(a, i, j); i++; }
    }
    swap(a, i, hi);
    quickSort(a, lo, i - 1);
    quickSort(a, i + 1, hi);
}
WHY QUICKSORT CAN HIT O(n²)

This implementation always picks the last element as the pivot. Feed it an already-sorted array and every partition splits into "everything" and "nothing" — n recursive levels, each doing O(n) work, for O(n²) total. Randomizing the pivot (or using median-of-three) makes that worst case astronomically unlikely in practice.

Searching

Halving the search space every step is the cheapest speedup available — but only if the data is sorted, and only if the boundary logic is exactly right.

The invariant that makes it work

At every step, the target — if it exists — is guaranteed to still be somewhere between lo and hi. Comparing the midpoint either finds it, or proves it must be strictly in the left half or the right half, letting you discard the other half entirely without ever looking at it.

BinarySearch.java
int binarySearch(int[] a, int target) {
    int lo = 0, hi = a.length - 1;
    while (lo <= hi) {
        int mid = lo + (hi - lo) / 2;   // avoids overflow vs (lo+hi)/2
        if (a[mid] == target) return mid;
        else if (a[mid] < target) lo = mid + 1;
        else hi = mid - 1;
    }
    return -1;
}
THE CLASSIC OFF-BY-ONE TRAP

If lo or hi isn't updated past mid when it should be, the loop can spin forever on a two-element window. Writing lo = mid + 1 / hi = mid - 1 (never plain mid) is what guarantees the window shrinks every iteration.

Variant — search in a rotated sorted array

A rotated array like [4,5,6,7,0,1,2] isn't fully sorted, but one half of any given window always is. At each midpoint, check which half is properly ordered, then check whether the target could possibly fall within that ordered half's range — if not, it must be in the other half. This keeps the same O(log n) halving even though the array isn't strictly sorted.

Recursion & Backtracking

Backtracking is depth-first search over a tree of decisions: try a choice, recurse as if it were final, then undo it and try the next one as though it never happened.

Backtrack.java
void backtrack(List<Integer> path /* , state */) {
    if (/* base case */) {
        result.add(new ArrayList<>(path));
        return;
    }
    for (/* each choice */) {
        path.add(choice);            // make the choice
        backtrack(path /* , ... */);
        path.remove(path.size() - 1); // undo it — this line is the part people forget
    }
}

Worked example — subsets of [1, 2, 3]

1At index 0, branch: include 1, or skip it. Both branches recurse independently.
2Including 1: at index 1, branch again on 2 → paths [1,2] and [1].
3Each of those branches on 3 → [1,2,3], [1,2], [1,3], [1].
4Skipping 1 at the start mirrors the same branching for 2 and 3 alone, producing the remaining four subsets — 8 total, because each of 3 elements independently doubles the branch count: O(2ⁿ).
WHY "PRUNING" MATTERS

The decision tree is often exponential by nature (subsets, permutations, N-Queens). Pruning — abandoning a branch as soon as it's provably invalid, like a queen attacking another in N-Queens — doesn't change the worst-case bound, but it's the difference between a backtracking solution finishing in milliseconds versus not finishing at all.

Dynamic Programming

Recursion with a memory. If you're solving the exact same subproblem more than once, you're paying for it twice — DP is just refusing to pay twice.

The two requirements

DP only applies when a problem has overlapping subproblems (the same smaller calculation gets needed repeatedly) and optimal substructure (the best overall answer can be built from the best answers to its subproblems). Fibonacci is the simplest example: fib(5) needs fib(3) twice if computed naively.

Memoization vs. tabulation

ApproachDirectionTrade-off
Memoization (top-down)Recursion + cacheOnly computes states actually needed; pays recursion call overhead
Tabulation (bottom-up)Iterative table fillNo recursion overhead, but fills every state, even unused ones
Memo.java
Map<Integer, Long> memo = new HashMap<>();
long fib(int n) {
    if (n <= 1) return n;
    if (memo.containsKey(n)) return memo.get(n);
    long result = fib(n - 1) + fib(n - 2);
    memo.put(n, result);
    return result;
}

Worked example — 0/1 knapsack, building the table

State: dp[i][w] = best value achievable using the first i items with capacity w. Transition: for each item, either skip it (dp[i-1][w]) or take it if it fits (dp[i-1][w - weight] + value) — take whichever is larger.

1Base case: dp[0][w] = 0 for all w — zero items, zero value, regardless of capacity.
2For each item i and each capacity w, fill in dp[i][w] = max(dp[i-1][w], dp[i-1][w-weight[i]] + value[i]) — but only consider the second option if weight[i] ≤ w.
3The answer ends up sitting in dp[n][capacity] — the best value using all items and the full capacity.
4Because each cell only depends on the row above, this can be optimized down to a single rolling 1D array — O(capacity) space instead of O(items × capacity).

Other classic patterns: longest common subsequence, longest increasing subsequence, coin change, edit distance — all follow the same five-step process from [00]'s sibling: define state, base case, transition, order, then optimize.

Greedy Algorithms

Take the best option available right now, commit, and never reconsider. It's the cheapest possible strategy — when it's actually correct.

Greedy only works when the problem has the greedy choice property: a locally optimal choice never closes off the path to a globally optimal solution. Proving this usually relies on an "exchange argument" — showing that any solution which deviates from the greedy choice can be rearranged into one that's at least as good.

A CASE WHERE GREEDY FAILS

Coin change with denominations {1, 3, 4}, target 6: greedy grabs the largest coin that fits each time — 4, then 1, then 1 — 3 coins. But 3 + 3 reaches the same target in 2. Greedy isn't wrong by accident here; this denomination set simply isn't "canonical," so the greedy choice property doesn't hold. The general coin change problem needs DP (see [12]) precisely because it can't assume canonical coins.

Where it does work: activity selection / interval scheduling (always pick the interval that finishes earliest), Huffman coding, fractional knapsack, jump game.

Two Pointers & Sliding Window

Two indices walking through the same array do the work of nested loops in a single pass — as long as the problem lets you reason about a contiguous range incrementally.

Fixed vs. variable windows

A fixed window (e.g. "max sum of any 3 consecutive elements") slides one position at a time, dropping the leftmost element and adding one on the right. A variable window (e.g. "longest substring without repeats") grows on the right and only shrinks from the left when some condition is violated — both pointers still only ever move forward, which is what keeps the whole thing O(n) instead of O(n²).

SlidingWindow.java
int left = 0, maxLen = 0;
Set<Character> seen = new HashSet<>();
for (int right = 0; right < s.length(); right++) {
    while (seen.contains(s.charAt(right))) {
        seen.remove(s.charAt(left));
        left++;
    }
    seen.add(s.charAt(right));
    maxLen = Math.max(maxLen, right - left + 1);
}

Worked example — "abcabcbb"

1right walks a, b, c — all new, window grows to "abc", length 3.
2Next char is a, already in the window. Shrink from the left — drop a — until a is no longer in the set.
3Window becomes "bca", length still 3. Continue rightward, repeating the same expand/shrink rhythm.
4The maximum length seen across the whole pass — 3 — is the answer, found in one O(n) sweep instead of checking every substring.

Bit Manipulation

The operations a CPU does natively, with no loop required — a handful of identities cover almost every bitwise interview question.

Bits.java
a & (a - 1);              // clear lowest set bit
a & 1;                     // check if odd
Integer.bitCount(a);       // count set bits
a ^ a == 0;                // XOR self-cancels

Why a & (a - 1) clears the lowest set bit

Subtracting 1 flips every trailing zero to a 1 and the lowest set bit to 0 (think of borrowing in subtraction — the borrow ripples through the trailing zeros). ANDing that with the original number keeps everything above the lowest set bit identical, but the lowest set bit itself becomes 0 — cleared in one operation, no loop needed.

Why XOR is the "find the unique number" trick

XOR has two properties that make it almost magical for this: x ^ x = 0 (a number cancels with itself) and XOR is commutative and associative (order doesn't matter). XOR every number in an array where everything appears twice except one — all the pairs cancel to 0, and 0 XORed with the lone number just returns that number. One pass, O(1) extra space.

Input Size → Complexity Guide

A sanity check, based on a rule of thumb that a typical machine handles roughly 10⁸ operations per second — so a 1–2 second time limit caps you at around 10⁸–10⁹ total operations.

n ≤ 10
O(n!) / O(2ⁿ·n)
n ≤ 20
O(2ⁿ) — bitmask DP
n ≤ 500
O(n³)
n ≤ 10,000
O(n²)
n ≤ 1,000,000
O(n log n) or O(n)
n > 1,000,000
O(n) or O(log n)

If n ≤ 20 shows up in a problem's constraints, that's almost always a deliberate hint that an exponential solution is expected and intended.
